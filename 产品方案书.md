# Codex++ 产品方案书

## 一句话定位

Codex++ 是面向普通用户的 Codex 增强客户端：用户完成购买或管理员开通权益后，登录 Codex++ Manager，客户端自动完成 Codex 安装检测、账号授权、设备绑定、托管供应商写入和一键启动。

本文档只描述产品方案和运营口径，不承诺固定价格、固定模型、固定额度或固定续费权益。所有销售内容、套餐权益、模型开放范围、额度策略、倍率、限流、续费入口和功能开关都以后台配置中心发布的当前配置版本为准。

## 目标用户

- 不熟悉 API、Base URL、模型协议和供应商配置的普通 Codex 用户。
- 希望统一售卖、开通、限额、审计和售后定位的运营管理员。
- 需要保留高级手动供应商配置，但希望默认路径更简单的进阶用户。

## 产品目标

- 用户购买或被管理员开通后，登录即可启动 Codex。
- 客户端自动写入 `Codex++ Cloud` 托管供应商，不要求普通用户填写 API Key、Base URL 或模型协议。
- 旧用户已有手动供应商配置必须保留，登录、刷新、退出和迁移都不能删除。
- 后台可以配置套餐、模型、额度、功能开关和续费文案，客户端无需发版即可体现。
- 网关在服务端强制执行权益、余额、模型权限、设备状态和限流策略，客户端只展示结果和行动建议。

## 用户主流程

1. 购买或开通权益
   - 用户通过官网、后台测试订单、兑换码或运营人工开通获得权益。
   - 具体价格、套餐名、模型包、额度和续费入口由后台配置版本决定。

2. 安装 Codex++ Manager
   - 用户下载当前发布包并完成安装。
   - 首次打开时，客户端检查 Codex 是否安装、配置目录是否可写、网络和本机权限是否可用。

3. 登录和授权
   - 生产路径优先使用浏览器授权。
   - 桌面端显示授权 URL 和 6 位确认码，不展示 `poll_token`、访问令牌或刷新令牌。

4. 自动配置
   - 客户端调用 `GET /api/v1/client/bootstrap` 拉取当前用户的最小运行快照。
   - 快照包含托管网关地址、用户可用状态、默认展示模型、功能开关、续费/修复动作和状态文案。
   - 客户端只写入或更新 `Codex++ Cloud`，不覆盖旧手动供应商。

5. 启动 Codex
   - 首页显示服务状态、设备状态和启动入口。
   - 用户点击启动后，Manager 交给 launcher 启动 Codex 并使用托管供应商完成请求。

6. 完成第一条任务
   - 新手用户先选择项目目录，再使用任务模板开口。
   - 推荐第一条任务文案：`帮我阅读这个项目，说明启动方式，并找出首页按钮点击无反应的可能原因。`
   - 客户端提示不要粘贴密码、支付信息、真实密钥或其他敏感内容。

## 管理员运营方案

管理员在后台配置中心维护可变销售和运行策略：

- 套餐：套餐名称、有效期、权益说明、续费入口、可用模型组、设备策略和功能开关。
- 模型：展示名称、真实路由、默认模型、上下架状态、回退策略和模型不可用提示。
- 额度：余额策略、每日额度、RPM、TPM、并发、预扣、实际结算和失败回补规则。
- 用户权益：订阅状态、余额账本、设备绑定、Key 摘要、测试开通和人工修正。
- 发布治理：草稿、校验、灰度范围、配置版本、发布时间、操作者、回滚点和审计记录。

管理员文案必须避免把价格、模型或额度写死在客户端安装包里。运营侧需要修改销售内容时，应先发布新的后台配置版本，再通过 bootstrap 快照让客户端刷新展示。

## 常见状态与用户文案

| 状态 | 用户看到 | 推荐动作 | 执行来源 |
| --- | --- | --- | --- |
| 未购买或未开通 | 当前账号还没有可用权益 | 前往购买、兑换或联系管理员开通 | 后台权益状态 |
| 套餐过期 | 服务已暂停，续费后可恢复 | 打开当前配置版本提供的续费入口 | 后台套餐有效期 |
| 余额不足 | 当前额度不足以继续请求 | 续费、充值或等待额度刷新 | 网关余额策略 |
| 设备撤销 | 这台设备已被撤销 | 重新绑定设备或联系管理员 | 后台设备状态 |
| 模型不可用 | 当前模型暂不可用或未开放 | 切换到后台配置允许的模型 | 模型配置和网关拒绝 |
| 网关不可用 | 服务暂时不可达 | 稍后重试或导出诊断信息 | Data Plane 健康状态 |
| 本地配置失败 | 托管供应商写入失败 | 一键修复、检查权限或导出日志 | Client Runtime |

## 工业级 v2 架构

Codex++ 按 `ARCHITECTURE.md` 的四层边界组织：

- Control Plane：管理套餐、模型、价格、额度、功能开关、权益、设备、审计和配置发布。
- Data Plane：提供客户端 API 和模型网关，在请求路径强制执行权益、余额、模型权限、设备状态、限流和风控。
- Client Runtime：处理登录、设备、bootstrap 缓存、`Codex++ Cloud` 托管供应商、首页状态和本地日志脱敏。
- Platform Ops：负责契约测试、E2E、兼容迁移、安装包检查、可观测、灰度、回滚、烟测和发布裁决。

关键链路：

- 配置发布：`Admin Console -> Config Registry -> Policy Decision -> Bootstrap Snapshot -> Client Runtime`
- 请求执行：`Client Runtime -> Client API Gateway -> Policy Enforcement -> Model Router -> Model Provider`
- 用量闭环：`Data Plane -> Usage Event -> Metering -> Billing / Audit / Risk`

## 灰度、回滚、审计和对账

- 灰度：配置版本可以按用户、设备、套餐或环境范围逐步发布。客户端只消费命中的 bootstrap 快照。
- 配置回滚：发现套餐、模型、额度或功能开关配置错误时，管理员回滚到上一稳定配置版本，客户端下次刷新快照后体现。
- 后端回滚：服务端回滚必须保留契约兼容，不能让旧客户端拿到缺字段或不认识的状态码。
- 桌面端回滚：安装包回滚不得删除 `Codex++ Cloud` 之外的手动供应商配置。
- 审计：管理员操作、配置发布、权益变更、设备撤销、网关拒绝和人工修正都需要可追踪。
- 对账：支付、退款、补偿、预扣、实际结算和失败回补都需要进入用量和账务核对链路。

## MVP 与生产上线边界

Phase 1 MVP 目标是完成测试环境闭环：后台开通或测试支付、登录、bootstrap、写入 `Codex++ Cloud`、启动 Codex、完成一次请求并查到 usage/log。

Phase 2 生产上线需要额外完成公网部署、HTTPS、数据库和 Redis 生产配置、密钥管理、真实或沙盒支付回调、监控告警、备份恢复、安全合规、客服运营和生产 go/no-go 门禁。没有通过生产 go/no-go 前，不应把 Phase 1 MVP 当成可售卖生产产品发布。

## 发布验收口径

- 新用户购买或开通后，登录即可启动 Codex 并完成一次请求。
- 旧用户手动供应商配置不丢失。
- 管理员改套餐、模型、额度和功能开关后，客户端无需发版即可刷新体现。
- 网关拒绝未授权模型、余额不足、套餐过期、设备撤销和限流超限。
- 用量事件可用于用户账单、管理员查询、风控分析和支付对账。
- 发布说明必须包含配置版本、契约版本、兼容影响、回滚路径、已知风险和等待中的 E2E 证据。
