执行规则

这几条是多会话开发时最容易跑偏的地方，HTML 用来把边界钉清楚。

依赖并行 按模块 DAG 和文件归属并行，不再简单把同一数字目录全部同时开跑。
契约门禁 `00-contract` 必须产出 OpenAPI、schema、mock、状态错误和事件契约。
文件归属 路由、schema、setting service、gateway hooks、lockfile 和全局 UI 必须有单一 owner。
模型质量 核心实现和集成使用高能力 worker；轻量模型只做只读探索、日志整理和摘要。
后台不绑 UI 后台配置只表达运营规则，不依赖 Codex++ 页面组件或前端结构。
网关不信任客户端 模型权限和用量策略必须在 Sub2API 网关侧强制执行。

最终成品代码结构

这里展示最终工程应该形成的模块边界和文件组织，不展示具体源码。开发会话按这些目录认领任务，避免把后台运营规则写进客户端。

1. Contract & Versioning 接口、配置、事件、错误码都要版本化，支持兼容检查和回滚。
2. Control Plane 承载后台运营、权益、计费、策略决策、配置发布和审计。
3. Data Plane + Client Runtime 数据面执行策略，客户端只消费快照并写入托管供应商。
4. Platform Ops 可观测、风控、迁移、发布门禁、烟测和回滚方案独立成层。

最终验收

验收重点是“可运营、可并行、可集成”，不是只有文档目录看起来完整。

✓
两阶段路线第一阶段做本地/测试 MVP，第二阶段补服务器、域名、HTTPS、QA 验收、监控和回滚。
✓
契约门禁OpenAPI、schema、mock、状态错误、事件和存储决策完成后才派发实现。
✓
文件归属路由、网关、setting service、UI 入口和 lockfile 有明确 owner。
✓
模型质量核心实现与集成使用高能力 worker，轻量 worker 只做只读辅助。
✓
网关执行下架模型、套餐过期、余额不足和限流必须由网关拒绝。
✓
无需发版后台新增套餐、新模型、新策略后，客户端通过 bootstrap 自动承接。
这份 HTML 是 `codex-plus-dev-plan/` 的可视化索引，保留原计划含义：阶段目录顺序推进，同级任务并行，客户端与后台完全解耦。
