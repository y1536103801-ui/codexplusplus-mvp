# Codex++ 管理员运营手册

Status: final for docs/product-copy evidence
Audience: admin operators, support, release coordinator
Release evidence boundary: E2E, package, compatibility and business readiness remain separate lanes.

## 运营原则

- 可变销售内容必须由后台配置中心管理，包括价格、套餐、模型、额度、倍率、限流、续费入口和功能开关。
- 客户端只消费 bootstrap 快照，不作为权益、余额、模型权限或设备状态的执行来源。
- 后端和网关是唯一 enforcement 来源。
- 每一次配置、权益、设备和人工修正都必须可审计、可追踪、可回滚。

## 配置中心操作

### 套餐和价格

- 维护套餐展示名、销售文案、有效期、续费入口和功能说明。
- 维护模型组、设备数量、额度策略、限流策略和并发策略。
- 发布前校验配置 schema、默认值和兼容字段。

### 模型和路由

- 维护展示模型到真实供应商模型的映射。
- 为默认模型、下架模型和回退模型设置可解释状态。
- 模型权限必须由网关在请求路径执行，客户端展示只用于引导。

### 额度和用量

- 维护余额、日额度、RPM、TPM、并发、预扣、实际结算和失败回补规则。
- 用量事件应可关联用户、设备、模型、请求 ID、配置版本和拒绝原因。
- 支付、退款、补偿和人工修正进入对账链路。

### 功能开关

- 管理高级供应商配置、安装辅助、Codex 增强、续费入口、诊断入口和灰度功能。
- 灰度范围可按用户、设备、套餐或环境配置。

## 发布流程

1. 草稿
   - 管理员创建配置草稿。
   - 校验套餐、模型、额度、功能开关和状态文案。

2. 灰度
   - 记录配置版本、操作者、发布时间、灰度范围和回滚点。
   - 观察 bootstrap 成功率、网关拒绝率、用量事件、支付对账和客服反馈。

3. 全量
   - 灰度稳定后扩大范围。
   - 发布说明同步配置版本、契约版本、兼容影响和已知风险。

4. 回滚
   - 配置错误时回滚到上一稳定配置版本。
   - 后端错误时回滚服务端版本并保持契约兼容。
   - 桌面端错误时回滚安装包，确认旧手动供应商仍保留。

## 审计与风控

- 审计记录覆盖配置发布、灰度、回滚、权益开通、权益修正、设备撤销、Key 轮换和风险处置。
- 公开文档不得披露内部风控规则、真实成本结构或上游供应商凭证。
- 客户端、后台和日志都不得展示完整 API Key、JWT、Authorization header 或 refresh token。

## 对账

- 支付回调需要幂等开通权益。
- 预扣和实际结算差异需要回补或修正。
- 退款、补偿和人工修正需要进入账本和审计。
- 对账查询至少应能按用户、订单、请求 ID、设备、模型和配置版本定位。

## 需要等待的 E2E 证据

- 购买或测试开通后登录并启动 Codex 的完整截图/日志。
- 网关拒绝未授权模型、余额不足、套餐过期和设备撤销的证据。
- 配置回滚后客户端刷新快照并恢复状态的证据。
- 用量事件与账务对账记录。
- 安装包安装、升级和回滚不删除旧手动供应商的证据。
