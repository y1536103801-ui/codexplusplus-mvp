系统架构

架构按 Control Plane、Data Plane、Client Runtime 和 Platform Ops 分层；客户端不内置共享 Key，也不保存上游真实密钥。

Client Runtime

  • 安装检测与修复
  • 登录、设备绑定、余额展示
  • 自动写入 Codex++ Cloud
  • 启动 Codex 并注入增强

Control Plane + Data Plane

  • 用户专属 API Key
  • 后台配置套餐、支付和兑换码
  • 计费、限流、并发控制
  • 账号池调度与后台看板

Platform Ops 与上游模型服务

  • OpenAI / Codex Responses
  • Claude、Gemini 等扩展渠道
  • 多账号容灾和失败切换
  • 按后台配置开放模型权限

两个项目运行流程

这部分对应文件夹里的两个真实项目：先把 Sub2API 跑成网关和后台，再把 Codex++ Manager 跑成桌面客户端，最后联调用用户侧 Key 走网关。

1. sub2api-main：先跑云端网关

Sub2API 是后端服务、管理后台、用户体系、套餐/余额和 OpenAI 兼容网关的来源。

# 本地开发 cd sub2api-main/backend go run ./cmd/server cd ../frontend pnpm install pnpm run dev
  • 后端默认服务端口是 `8080`；前端 Vite 默认端口是 `3000`。
  • 前端代理 `/api`、`/v1`、`/setup` 到后端，目标默认 `http://localhost:8080`。
  • 本地开发需要 PostgreSQL 15+ 和 Redis 7+；生产更适合用 Docker Compose。

2. CodexPlusPlus-main：再跑桌面客户端

Codex++ Manager 是 Tauri 管理工具；launcher 是静默启动入口，负责打开 Codex 并注入增强。

# 桌面端开发 cd CodexPlusPlus-main/apps/codex-plus-manager npm install npm run dev # 打包 npm run build
  • `npm run dev` 执行 `tauri dev`，并自动启动 Vite 到 `127.0.0.1:1420`。
  • `npm run build` 会先构建 `codex-plus-launcher`，再执行 Tauri 打包。
  • 运行时由 Manager 检测 Codex、写入供应商配置，再交给 launcher 启动 Codex。

两个项目联调链路

Step 1 启动 Sub2API 配置数据库、Redis、上游账号、套餐、余额和用户 API Key。
Step 2 启动 Codex++ Manager 桌面端进入登录、检测、自动配置和启动流程。
Step 3 拉取托管配置 目标产品需要把 Sub2API 能力包装成 `/api/v1/client/bootstrap` 等客户端接口。
Step 4 写入 Codex++ Cloud 客户端生成托管供应商：Base URL 指向 Sub2API 的 `/v1` 网关，Key 使用用户侧 Key。
Step 5 请求闭环 Codex 请求进入 Sub2API，后台完成鉴权、计费、限流、账号池调度和用量回写。

验收标准

验收重点不是“能配置”，而是普通用户全程不需要配置。

✓购买后可启动新用户登录后能完成一次可用请求。
✓无 API 配置全程无需填写 Base URL、Key 或模型协议。
✓状态可理解余额不足、过期、禁用都有明确原因和动作。
✓支付能恢复续费成功后刷新客户端即可恢复服务。
✓后台可运营管理员能查看用户、订单、设备、Key 和用量。
✓高级能力保留手动供应商配置仍可用，但不出现在默认路径。
默认决策：保留登录体系；不在安装包内置共享 Key；客户端只保存 Sub2API 用户侧 Key；第一版优先完成“买完即用”闭环。
