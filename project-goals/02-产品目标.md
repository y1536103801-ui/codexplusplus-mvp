产品目标
========

目标一：让普通用户只在桌面客户端实际使用服务
------------------------------------------

Codex+++ 的普通用户入口必须是桌面客户端。

桌面客户端的目标是把复杂技术过程封装成：

1，登录。
2，状态展示。
3，客户端内选择 Token 充值项并提交充值请求。
4，Codex+++ token 余额展示。
5，本机自动准备。
6，一键启动 Codex。
7，Codex 账号余额不足或当前 API Key 不可用时由后端 gateway 切换可用 API Key 或等价可用路由。
8，异常时给出中文提示。
9，默认简洁模式和可手动进入的高级模式。

普通用户不得通过浏览器页面登录、查看余额或交易记录、运行 Codex 或处理账号状态。已登录用户的充值请求必须在桌面客户端内发起，并由管理员在 Web 管理后台确认。

官网是购买前入口，只负责项目介绍、Windows 与 macOS 客户端下载、付费 Token 套餐展示和新账号购买申请。官网申请由管理员人工联系和确认收款，不等同于公开注册、自动支付或普通用户 Web 自助业务。

目标二：让管理员只使用 Web 管理后台运营
---------------------------------------

Codex+++ 的受保护浏览器管理界面只面向管理员；官网是与管理后台分离的公开入口，不得暴露管理功能。

Web 管理后台的目标是让管理员完成系统运营所需操作：

1，初始化首个管理员。
2，管理员登录。
3，首轮更改账户密码。
4，管理普通用户。
5，管理 Token 充值项。
6，确认普通用户的充值请求或为普通用户调整 Codex+++ token。
7，管理普通用户的 Codex+++ token 余额、最近充值状态和可用状态。
8，管理设备。
9，管理 Codex 账户号池、账户凭据和后端生成的 API Key。
10，查看用量、token 流水和审计日志。
11，查看待确认充值、可用号池账号和今日 token 用量的运营概览。
12，处理官网账号购买申请，记录联系状态，并在创建或绑定普通用户账号后完成初始 Token 入账。

首轮更改账户密码是管理员初始化后的流程步骤，不是密码内容限制。管理员密码内容由管理员自行决定；系统不得强制非空、长度、复杂度或安全强度提醒。

Web 管理后台不得变成普通用户自助业务；购买申请中的联系方式只向已认证管理员解密展示。

目标三：自建 Codex+++ 后端
-------------------------

Codex+++ 需要自有后端。

这个后端的目标是统一承担：

1，管理员认证和权限。
2，普通用户认证。
3，设备登记和设备状态。
4，Token 充值项、Codex+++ token 余额、最近充值状态和用户可用状态。
5，token 预留、扣减、释放、退款和对账。
6，导入并维护服务端 Codex 账户号池。
7，将号池资源转成后端可调度的 API Key。
8，gateway 根据 API Key 对应的 Codex 账号余额、API Key 状态、策略和风控结果选择可用上游账户并转发请求。
9，后端 gateway 发现 Codex 账号余额不足或当前 API Key 不可用时，切换到下一枚可用 API Key 或等价可用路由。
10，Codex+++ token 余额不足时，后端不得通过切换 API Key 继续消耗服务，客户端不得主动触发充值引导、联系管理员或 API Key 切换动作；后端不再提供后续 Codex 账号、API Key 或等价可用路由，由 Codex 自身呈现不能继续使用的提示。
11，保护 Codex 账户凭据和 API Key，避免普通用户看到或手动配置。
12，用量、token 流水、幂等记录和审计。
13，向官网提供最小公开配置、接收限流后的账号购买申请，并加密保存购买者联系方式。

sub2api 必须作为机制参考，重点参考它“导入 Codex 上游账户形成号池，把号池资源转成可调度 API Key，按 Codex 账号余额和 API Key 状态切换可用 API Key”的机制；sub2api 不得作为 Codex+++ 的运行时后端。

目标四：复用 Codex++ 的合适本机能力
----------------------------------

Codex+++ 保留复用 Codex++ 中适合桌面端的本机能力。

必须复用或参考的能力包括：

1，Tauri 桌面壳。
2，本机 Codex 检测。
3，受控配置清理。
4，启动流程。
5，Codex 官方登录态读取。
6，更新能力。
7，脱敏后的本机诊断能力。

这些能力用于服务普通用户的一键使用体验，不得把旧 Codex++ 的高级工作台直接暴露给普通用户。

参考项目落点
------------

Codex++ 和 sub2api 是当前产品目标的明确参考来源。

必须参考 Codex++ 的目标包括：

1，桌面客户端形态。
2，Tauri 桌面壳。
3，本机 Codex 检测。
4，受控配置清理。
5，启动 Codex 的本机流程。
6，Codex 官方登录态读取。
7，客户端更新。
8，脱敏后的本机诊断。

Codex+++ 参考 Codex++ 是为了继承适合普通用户一键使用的本机能力，不是把旧 Codex++ 高级工作台继续包装成普通用户产品。

必须参考 sub2api 的目标包括：

1，导入 Codex 上游账户形成号池。
2，维护 Codex 账号状态、余额状态和可路由状态。
3，把号池资源转成后端可调度 API Key。
4，维护 API Key 与上游 Codex 账号或等价路由的关系。
5，按 Codex 账号余额、API Key 状态、策略和风控结果选择可用 API Key。
6，通过 gateway 承接请求、鉴权、调度、上游调用和受控响应。
7，记录 token、用量、幂等、风控和审计。

Codex+++ 参考 sub2api 是为了定义后端机制目标，不是把 sub2api 当作运行时后端或普通用户浏览器账户中心继续改造。Codex+++ 官网购买申请属于自有销售入口，不复用 sub2api 的公开注册、支付或账户页面。

目标五：保护密钥和内部细节
-------------------------

Codex+++ 让 Codex 账户凭据只存在于服务端受控范围。API Key 由后端生成、分配、校验和切换。

普通用户不得看到或手动输入 Codex 账户凭据、API Key、proxy、endpoint、base_url、内部 gateway 地址、Authorization header、令牌、密钥片段或上游错误原文。

管理员可以导入 Codex 账户或替换账户凭据，但后台列表、审计日志和普通响应不得展示账户凭据原文、API Key 原文或可稳定识别同一凭据的摘要。
