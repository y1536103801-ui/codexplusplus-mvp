扣费与用量
==========

目标
----

保证 Codex+++ token 余额扣减可信、可对账。

参考项目
--------

本模块必须参考 sub2api 中与额度、用量、请求记录、幂等和审计相关的机制。

参考 sub2api 的结果必须落到 Codex+++ 自己的 token 余额和 Token 充值项模型。Codex+++ token 余额是普通用户提交充值请求并经管理员确认入账后，在 Codex+++ 内可消费的 token，不得和 Codex 账号余额混写。

首版 token 按 Codex 自身 token 用量和 Codex 规则扣减。

必须做到
--------

1，请求前预留 Codex+++ token。
2，请求成功后提交实际扣减。
3，请求失败后释放预留。
4，重复 request_id 不得重复扣费。
5，写入用量和 token 流水。
6，管理员可查看用量和 token 流水。
7，扣费依据必须来自后端可验证的 Codex 用量结果，不得直接使用客户端自报 token。
8，客户端只是启动器，不负责自行计算 token、模型倍率或计费规则。
9，流式请求中断、超时或上游没有返回 usage 时，首版按 Codex 自身规则和可验证结果处理。
10，普通用户可以在高级模式看到自己的 token 收入和消耗流水。
11，普通用户简洁模式显示最近 7 日 token 用量迷你趋势图。
12，高级模式的 token 收入和消耗流水采用最近 30 日正负柱状图加流水表。
13，流水表字段为时间、类型、token 变动数量、变动后余额和来源。

不得做到
--------

1，用客户端自报用量直接扣费。
2，在 Codex+++ token 余额不足时继续扣成负数。
3，把 Codex 账号余额当作普通用户余额。
4，自立门户设计独立于 Codex 规则之外的首版计费规则。
5，隐藏普通用户账户的 token 收入和消耗。
