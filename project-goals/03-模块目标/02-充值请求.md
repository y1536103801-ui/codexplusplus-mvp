充值请求
========

目标
----

让普通用户登录后在桌面客户端内选择 Token 充值项并提交充值请求。

必须做到
--------

1，展示管理员创建并启用的 Token 充值项。
2，让普通用户在桌面客户端内提交充值请求。
3，管理员在 Web 管理后台确认充值请求或在普通用户详情中手动调整 token 后，形成 Codex+++ token 余额。
4，充值确认结果能刷新到普通用户简洁模式和高级模式。
5，充值失败时显示中文原因和下一步动作。
6，首版充值流程不接真实自动支付通道。
7，普通用户发起充值请求时只点击“申请充值”，不填写备注、金额、token、原因或其他附加内容。
8，充值请求在管理员确认前必须记录申请的 Token 充值项、充值项对应 token、提交时间、当前状态和状态流转。
9，管理员确认充值请求时按 Token 充值项规则入账 token，不在确认动作中临时修改到账 token。
10，管理员拒绝充值请求时不强制填写拒绝原因。
11，充值请求首版状态只使用 pending、approved、rejected、cancelled。
12，普通用户客户端展示 Token 充值项时显示充值项名称、人民币价格和对应 token。
13，Token 充值项价格允许为 0，价格为 0 时显示为 free。
14，普通用户点击申请充值后，客户端显示“等待管理员确认”状态。
15，价格大于 0 的 Token 充值项允许重复申请；每次确认后都按充值项规则增加 token 余额。
16，价格为 0 的 free Token 充值项仅供查看，不提供普通用户端申请动作。
17，普通用户可以在高级模式查看历史充值请求记录。
18，管理员确认充值请求后，普通用户客户端只刷新 token 余额，不弹出成功提示。
19，历史充值请求记录只显示 Token 充值项、状态、提交时间和确认时间。
20，历史充值请求记录不显示确认管理员。
21，已确认充值请求不允许管理员后台撤销；需要修正时通过普通用户详情中的 token 调整形成新流水。
22，普通用户历史充值请求需要分页。
23，历史充值请求每页默认显示 10 条。
24，普通用户 Token 充值项区域展示所有已启用 Token 充值项；价格大于 0 的 Token 充值项显示“申请充值”动作，价格为 0 的 free Token 充值项不显示申请动作。
25，简洁模式中的充值入口登录后常驻显示，不仅在 token 不足时出现。
26，待确认充值页面列表字段为用户账号、Token 充值项、充值项 token、提交时间、状态和操作。
27，待确认充值页面操作只包含单条确认和单条拒绝。
28，管理员拒绝充值请求后，普通用户历史充值请求状态显示“已拒绝”。

不得做到
--------

1，提供普通用户 Web Token 充值项页。
2，提供普通用户 Web 支付页。
3，提供独立于桌面客户端的普通用户交易页面。
4，提供独立于桌面客户端的普通用户记录页面。
5，提供独立于桌面客户端的普通用户账户页面。
6，提供普通用户自助充值。
7，要求普通用户填写充值备注、金额、token 或原因。
8，允许管理员后台撤销已确认充值请求。
9，允许 free Token 充值项由普通用户反复申请。
10，在普通用户历史充值请求中显示确认管理员。
11，在待确认充值页面批量确认充值请求。
12，把管理员手动充值或调整 token 放在待确认充值页面。
13，把价格大于 0 的 Token 充值项锁定为一次性领取。
