号池管理
========

目标
----

让管理员管理可供 Codex+++ 调度的上游 Codex 账号。

参考项目
--------

本模块必须参考 sub2api 的号池机制，重点参考导入 Codex 上游账户、账号分组、账号凭据维护、余额状态检查、可用性检查、停用启用、风控标记和审计记录。

参考 sub2api 的结果必须落到 Codex+++ 自有后端的号池管理能力，不得把 sub2api 当作 Codex+++ 后端继续改造。

sub2api 支持多种上游账号类型，Codex+++ 首版只围绕 Codex 上游账号建立干净号池，不继承多供应商兼容形态。

必须做到
--------

1，导入 Codex 账号到号池。
2，保存 Codex 账号凭据。
3，检查 Codex 账号是否可用。
4，识别 Codex 账号余额状态。
5，停用和启用 Codex 账号。
6，关键操作写入审计记录。
7，首版 Codex 上游账号字段包括账号名称、账号分组、来源类型、凭据类型、授权状态、access_token、refresh_token、Codex `auth.json` 快照、临时密码、token_type、expires_at、可选 email、可选 subscription_tier、entitlement_status、账号状态、余额状态、风控状态、最近授权错误和最近检查时间。
8，敏感字段必须加密保存，不得在后台列表、审计、日志或普通响应中展示原文。
9，管理员后台支持单个和批量导入。输入格式包括 ChatGPT session JSON、邮箱密码文本、邮箱密码 JSON、Codex `auth.json`、Token JSON、Sub2API 账号备份和现有纯 Token 文本。ChatGPT session JSON 必须由后端提取最小授权材料并加密保存，不保存头像等无关会话资料；没有 refresh token 时标记为短期授权，同一账号重新导入时更新原记录和已有内部 API Key。
10，号池账号列表显示分组。
11，号池账号不可用只在后台显示，不通知普通用户。
12，正式账号的调度状态显示“可用”和“不可用”；候选账号另行显示待授权、等待完成授权、授权失败和已授权状态。
13，号池账号分组首版由管理员手动维护。
14，号池账号列表显示最近检查时间。
15，运营概览中的可用号池账号数量点击后进入号池账号列表，并自动筛选为可用账号。
16，邮箱密码导入后创建待授权候选账号。候选账号不得生成内部 API Key，不得进入 Gateway 调度。
17，候选账号密码使用服务端密钥加密暂存。授权成功后在同一账号记录上写入授权材料并清除密码密文。
18，后台授权支持 Codex app-server 浏览器登录和 device-code 登录。两种方式均使用隔离 `CODEX_HOME`，完成后读取同目录 `auth.json` 并清理临时目录。
19，浏览器登录通过 `account/login/start` 的 `chatgpt` 类型取得 `authUrl`。远程部署时由管理员粘贴浏览器最终产生的完整 localhost 回调地址，后端只允许 `http://localhost`、`http://127.0.0.1` 或 `http://[::1]` 的 `/auth/callback`，并校验 OAuth state 后转发到同一后端内的 app-server 监听端口。
20，device-code 登录通过 `account/login/start` 的 `chatgptDeviceCode` 类型取得 `verificationUrl`、`userCode` 和 `loginId`。
21，后台不保存 OpenAI OAuth client id、token endpoint 或授权码交换实现。浏览器和 device-code 登录均由 Codex app-server 管理。
22，app-server 授权结果必须包含 refresh token。缺少 refresh token 的结果不得进入自动刷新维护链。
23，授权结果绑定候选账号时必须校验账号邮箱一致性。邮箱不一致时保留候选账号并记录授权失败状态。
24，Codex `auth.json`、Token JSON 或 Sub2API 备份中已包含有效授权材料的账号可直接进入正式号池并生成内部 API Key。
25，账号来源格式只影响导入适配，不扩展公开 API 路由或 Gateway 协议。
26，号池导入、上游授权、凭据保存、刷新和内部 API Key 生成只属于后端与管理员 Web。普通用户 Windows 与 macOS 客户端均不接收号池邮箱密码、上游授权材料或号池管理动作。
27，管理员可以导出可重新导入的完整号池 JSON。导出包含恢复所需的上游授权材料和待授权密码，必须作为敏感文件处理；不得包含 Codex+++ 内部 API Key、Key hash、公开前缀或凭据指纹，导出动作必须审计。
28，浏览器授权、device-code 授权和添加账号面板都提供明确关闭按钮。关闭授权面板时取消后端授权任务并清理隔离 app-server；关闭添加账号面板时不得提交未保存内容。
29，号池正式账号列表以使用情况为优先，行内显示当前使用用户。当前使用包括进行中的 Gateway 请求，以及基于最近一次成功 Gateway 归属、由运行中客户端持续续租的用户；客户端停止 Codex 或心跳超时后显示最近使用者。客户端不得提交、选择或猜测上游账号 ID。同时显示该账号当日经 Codex+++ 路由的 token、上游剩余百分比或 credits 和账号状态。
30，没有当前使用用户时显示绿色“空闲”标识；最近使用用户和最近一次会话的使用时间段只在鼠标悬浮该标识时显示，不常驻占用表格空间。
31，后端按北京时间每天 09:00 自动检查全部已授权号池账号；服务错过执行时间后在当日恢复时补检尚未完成当天检查的账号。自动检查与手动检查使用同一检查逻辑，并以 system 角色写入审计。
32，号池账号增加最多 500 个字符的管理员备注。正式账号列表显示备注列，每行操作可新增、修改和清空备注；备注随号池备份导出和恢复，但备注正文不得写入审计。
33，客户端已启动但尚未产生首个 Gateway 任务时，后台显示该普通用户“使用中、待首次任务分配”，但不得把用户虚假挂到任意号池账号；首个真实任务完成路由后才在对应号池账号行显示当前使用用户。
34，手动或自动检查执行期间，号池账号行必须显示“检查中”并禁止重复触发。检查失败后必须显示经过安全分类的失败原因，例如认证失败、额度限制、频率限制、检查超时或检查服务不可用；不得把原始异常、路径、令牌或授权内容返回页面。

不得做到
--------

1，把 sub2api 当作 Codex+++ 后端继续改造。
2，把 Codex 账号凭据展示给普通用户。
3，把 Codex 账号凭据展示在普通响应中。
4，继承 sub2api 的多供应商兼容形态。
5，把号池账号不可用通知给普通用户。
6，以明文保存、返回或记录号池账号密码。
7，把待授权候选账号生成 API Key 或加入 Gateway 调度。
8，让普通用户客户端发起或完成上游账号授权。
9，自行拼接 OpenAI OAuth 授权 URL。
10，自行维护 OpenAI OAuth client id。
11，自行维护 OpenAI OAuth token endpoint 或授权码交换实现。
12，把 Sub2API 的公开路由、页面、数据库结构或多供应商模型作为兼容层引入 Codex+++。
