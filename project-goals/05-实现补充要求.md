参考项目实现边界
----------------

联网查找确认的 Codex++ 默认来源是 `https://github.com/BigPizzaV3/CodexPlusPlus`。该项目定位为面向 Codex App 的外部增强启动器和管理工具，包含静默启动入口和 Tauri 管理工具，重点能力包括启动、检查、修复、更新、配置中转注入、增强功能和用户脚本管理。

联网查找确认的 sub2api 默认来源是 `https://github.com/Wei-Shaw/sub2api`。该项目定位为 AI API gateway 平台，核心能力包括多上游账号管理、平台生成 API Key、鉴权、计费、负载均衡、请求转发、智能调度、并发控制、限流和用量统计。

Codex++ 的参考重点是桌面端能力：Tauri 桌面壳、本机 Codex 检测、受控配置清理、启动流程、更新和脱敏诊断。Windows 使用 Store/AppX，macOS 使用 OpenAI 官方签名应用包，不共享平台专属实现。

sub2api 的参考重点是后端调度机制：上游账号号池、平台生成 API Key、鉴权、计费、负载均衡、请求转发、智能调度、并发控制、限流、用量统计和审计。

Lago 作为 usage-based billing、prepaid credits 和 wallet 设计参考。Codex+++ 只参考其用量计量、预付余额和钱包建模思路，不引入 Lago 的支付、发票、催收、优惠券、税务、完整账务或多产品复杂能力。

参考两个项目不表示照搬界面、运行时代码、数据库表名或权限模型。Codex+++ 必须形成自己的桌面客户端、Web 管理后台、后端服务和共享契约。

Codex 官方定价调研
------------------

联网查找确认，Codex 官方定价已经是 token-based。Codex usage 按 API token usage 计算，主要区分 input tokens、cached input tokens 和 output tokens。

OpenAI Codex pricing 页面说明，API Key 场景只按 Codex 使用的 token 付费，并基于 API pricing。ChatGPT 计划和 Business/Enterprise 场景可能以 credits 表达，但底层仍按每百万 input、cached input 和 output token 映射。

当前官方 Codex rate card 中的示例包括：

1，GPT-5.5：每 1M input token 为 125 credits，每 1M cached input token 为 12.50 credits，每 1M output token 为 750 credits。
2，GPT-5.4：每 1M input token 为 62.50 credits，每 1M cached input token 为 6.250 credits，每 1M output token 为 375 credits。
3，GPT-5.4-Mini：每 1M input token 为 18.75 credits，每 1M cached input token 为 1.875 credits，每 1M output token 为 113 credits。
4，GPT-5.3-Codex：每 1M input token 为 43.75 credits，每 1M cached input token 为 4.375 credits，每 1M output token 为 350 credits。

当前 API pricing 中，`gpt-5.3-codex` 的标准价格为每 1M input token 1.75 美元、每 1M cached input token 0.175 美元、每 1M output token 14.00 美元；Priority 价格为每 1M input token 3.50 美元、每 1M cached input token 0.35 美元、每 1M output token 28.00 美元。

Codex+++ 产品层不得再使用旧的点数类展示单位。首版普通用户和管理员界面统一显示 token、token 余额、token 用量和 token 流水。若后端需要处理 OpenAI credits 或美元价格，只能作为内部结算映射，不得替代产品层的 token 表达。

充值项机制调研结论
------------------

联网调研后，Codex+++ 首版不再使用“固定权益档位”作为充值模型。

原因如下：

1，Codex 官方计费不是一次性买断能力，而是基于 input token、cached input token 和 output token 的持续消耗。
2，OpenAI Codex 和 ChatGPT 相关文档都把可追加使用量表达为 credits 或 add credits；不可表达为不可重复申请的固定权益。
3，usage-based billing 和 prepaid credits 的常见做法是用户先获得可消耗余额，再按实际使用持续扣减。
4，Token 充值项更像“给 token 钱包充值的商品”，不应承担订阅周期、固定权益、到期时间或不可重复申请语义。

因此，Codex+++ 首版采用以下模型：

1，普通用户只有 Codex+++ token 钱包余额。
2，管理员创建 Token 充值项，用于定义人民币展示价格和到账 token 数量。
3，普通用户在客户端选择 Token 充值项并提交充值请求。
4，管理员确认充值请求后，token 入账到普通用户 token 钱包。
5，价格大于 0 的 Token 充值项允许同一普通用户重复申请。
6，价格为 0 的 free Token 充值项只展示，不提供普通用户端申请动作；需要赠送时由管理员手动调整 token。
7，历史记录记录的是充值请求和 token 流水，不记录“一次性权益状态”。

当前默认技术选择
----------------

桌面客户端首版选择 Windows + Tauri + Rust + 前端界面实现，依据是 Codex++ 的 Windows/Tauri 桌面能力。

后端和 gateway 默认选择 Go，参考 sub2api 的 Go、Gin、Ent 实现方向。最终实现可按工程实际调整具体框架，但不得因此偏离 Codex+++ 的自有后端和干净契约目标。

Web 管理后台默认选择 Vue、Vite 和 TailwindCSS，参考 sub2api 的管理后台技术栈。最终实现可按工程实际调整具体 UI 组件库，但不得引入普通用户 Web 账户中心或自助业务。公开官网是独立入口，只承担介绍、下载和人工审核的账号购买申请。

数据库默认选择 PostgreSQL。Redis 默认作为缓存、队列、限流、调度和短期状态存储使用。

gateway 干净契约
----------------

Codex+++ gateway 首版必须是唯一、干净的自有协议契约。

不得暴露根级 `/v1/chat/completions`、`/v1/responses`、`/v1/messages`、Images 或其他 OpenAI、Anthropic、sub2api 历史路由。

允许保留唯一的 Codex 形态鉴权入口 `/api/codex/v1/models` 和 `/api/codex/v1/responses`。该入口属于后端受控适配能力，必须复用后端内部 gateway executor 的鉴权、余额检查、扣费、幂等、路由和审计逻辑，不得成为旧 OpenAI 兼容层或 fallback。Windows 与 macOS 客户端在无本机 Codex 登录态，或已有登录账号通过本后端号池归属校验时可以把该入口写入 Codex provider 配置；外部个人账号不得被覆盖。

不得保留旧路由、旧字段、旧页面、旧配置、旧兼容层或旧 fallback。发现旧内容时，应即时删除或迁移到 Codex+++ 唯一契约。

Codex+++ 客户端、后端和 gateway 之间只允许使用 Codex+++ 明确定义的单一契约。具体路径、字段和错误码由实现阶段决定，原则是唯一、干净、可运营，不作为产品呈现问题询问用户。

账号与密码
----------

普通用户账号由管理员创建。

管理员创建普通用户时，可以设置普通用户密码。普通用户密码内容由管理员决定，系统不得强制非空、长度、复杂度或安全强度提醒。

管理员创建普通用户时不要求设置初始 token。未设置初始 token 时，普通用户初始 Codex+++ token 余额为 0。

普通用户登录页不显示注册入口，不提供忘记密码或找回密码。

普通用户未知账号登录失败时，不得自动创建账号。

管理员可以在普通用户详情中重置普通用户密码。重置密码不得设置非空、长度、复杂度或安全强度限制。

管理员创建普通用户时不展示一次性初始密码。密码由管理员设置，保存后不再显示原文；如果管理员忘记已设置密码，只能在普通用户详情中重置。

充值请求与 token
---------------

普通用户在桌面客户端内选择 Token 充值项并提交充值请求。

管理员在 Web 管理后台查看充值请求，并确认入账。管理员也可以在普通用户详情中手动调整 token。

普通用户发起充值请求时只点击申请充值。客户端不得要求普通用户填写备注、金额、token、原因或其他附加内容。

充值请求在管理员确认前必须记录申请的 Token 充值项、充值项对应 token、提交时间、当前状态和状态流转。

充值请求首版状态只使用 `pending`、`approved`、`rejected`、`cancelled`。

管理员确认充值请求时按 Token 充值项规则入账 token，不在确认动作中临时修改到账 token。

管理员拒绝充值时不强制填写拒绝原因。

充值确认或手动调整成功后，普通用户的 Codex+++ token 余额增加。

管理员确认充值请求后，普通用户客户端只刷新 token 余额，不弹出成功提示。

价格大于 0 的 Token 充值项允许重复申请。每次确认后都按充值项规则增加 token 余额，不把充值项锁定为一次性领取状态。

普通用户可以查看历史充值请求记录。

历史充值请求记录只显示 Token 充值项、状态、提交时间和确认时间。

历史充值请求记录需要分页，每页默认显示 10 条。

已确认充值请求不允许管理员后台撤销。需要修正时，通过普通用户详情中的 token 调整形成新流水。

充值请求列表和详情首版不显示确认管理员。

管理员后台需要单独的待确认充值页面。运营概览中的待确认充值数字可以点击进入该页面。

管理员处理充值请求首版不做批量确认。

价格为 0 的 free Token 充值项仅供查看，不提供普通用户端申请动作。需要赠送 token 时，由管理员在普通用户详情中手动调整。

待确认充值页面列表字段为用户账号、Token 充值项、Token 充值项 token、提交时间、状态和操作。操作只包含单条确认和单条拒绝。

管理员拒绝充值后，普通用户历史充值请求状态显示“已拒绝”。

普通用户 Token 充值项区域展示所有已启用 Token 充值项。价格大于 0 的 Token 充值项显示“申请充值”，价格为 0 的 free Token 充值项不显示申请动作，停用 Token 充值项不显示为新充值选项。

简洁模式中的充值入口登录后常驻显示，不仅在 token 不足时出现。充值入口常驻不表示客户端主动引导充值，也不得在 token 不足时自动触发充值。

Codex+++ token 余额用于 gateway 请求前预留、请求成功后扣减、请求失败后释放和异常修正。

token 首版按 Codex 自身 token 用量和 Codex 规则扣减。客户端只是启动器，不负责自行计算 token、模型倍率或计费规则。gateway 必须以后端可验证的 Codex 用量结果为依据，不得使用客户端自报 token 直接扣费。

gateway 首版通过 Codex 官方 Responses 协议执行普通用户任务。后端使用所选上游账号的 access token 和 ChatGPT account id 请求 Codex Responses 端点，原样返回 SSE 工具事件和允许透传的 Codex 响应头，并只在完成响应包含可验证 usage 时写入用量和扣减 token。本机 Codex 执行 shell、文件和其他本地工具。后端 app-server 仅用于账号授权、凭据刷新和账号检查。不得保留桥接运行 URL、根级 OpenAI 兼容路由、sub2api 运行时路由或历史 fallback。

gateway 预留是后端内部请求状态，不作为普通用户 token 流水展示。只有上游返回可验证 usage 并完成实际扣减后，才写入普通用户可见的 token 消耗流水。重复 request_id 命中已完成请求时，只返回已有用量记录，不再次请求上游，也不再次扣费。

gateway 在上游账号余额不足、上游认证不可用、上游限流、上游路由配置不可用或上游服务不可用时，可以将当前上游账号标记为不可用并自动尝试下一条可用 API Key 路由。上游拒绝请求参数形态时，不作为账号切换理由。Codex+++ token 余额不足不得进入此切换逻辑。

流式请求中断、超时或上游没有返回 usage 时，首版按 Codex 自身规则和可验证结果处理，不自立门户设计额外计费规则。

token 不足行为
------------

Codex+++ token 不足时，客户端不主动触发任何动作。

客户端不得弹出充值引导，不得自动联系管理员，不得自动发起充值，不得自动切换 API Key，也不得把 token 不足解释成上游账号不可用。

余额不足时，后端不再提供后续 Codex 账号、API Key 或等价可用路由，由 Codex 自身呈现不能继续使用的提示。客户端不得额外显示“余额不足”提示。

普通用户 token 余额为 0 时仍允许登录客户端。余额不足只影响后续账号供给和继续消耗服务，不影响进入客户端查看状态、Token 充值项和记录。

普通用户客户端界面
------------------

普通用户登录成功后默认进入简洁模式。

简洁模式显示账号、最近充值状态、剩余 token 精确数字、最近 7 日 token 用量迷你趋势图、服务状态、启动状态和一个明确主动作。

服务状态区域中的 Codex 账号显示只读取本机 Codex 认证状态检测结果。已有 ChatGPT 账号信息时显示本机识别出的账号信息；只有 API-key 登录态时显示本机 `auth.json` 中 `OPENAI_API_KEY` 的脱敏值；本机无法识别出认证信息时显示“未识别”。该字段不得读取后端号池账号、后端会话 token、启动器登录账号或后端返回的展示账号字段。

简洁模式提供申请 Token 充值项入口和进入高级模式的按钮。已登录普通用户的充值入口仍然是客户端内充值请求；官网账号购买申请只面向尚未获得普通用户账号的访客，两类申请不得混用记录或状态。

高级模式面板参考 Codex++ 的管理工具分层：普通入口负责使用，高级面板承载记录、状态和脱敏诊断。Codex+++ 不复制旧 Codex++ 高级工作台。

高级模式显示 token 用量明细、token 收入和消耗流水、历史充值请求、设备状态、安全状态和脱敏诊断。

token 用量明细需要分页，每页默认显示 10 条。用量明细列表可用小型横向堆叠条表达 input token、cached input token 和 output token 的组成，但表格字段仍以数字为准。

历史充值请求放在高级模式中展示，每页默认显示 10 条。

token 收入和消耗流水采用两层呈现：上方显示最近 30 日 token 收入和消耗的正负柱状图，下方显示可核对的流水表。流水表字段为时间、类型、token 变动数量、变动后余额和来源。

普通用户高级模式的脱敏诊断只显示五类信息：本机 Codex 是否检测到、本机 Codex 是否正在运行、本机 Codex 版本、版本是否兼容和最近一次启动失败原因。不得展示诊断原文、路径细节、API Key、proxy、endpoint、base_url 或内部 gateway 配置。

普通用户可视化调研结论
----------------------

联网调研后，普通用户界面采用“首屏快速理解，高级模式精确核对”的可视化原则。

调研依据包括：

1，Nielsen Norman Group 的 dashboard 指南强调 dashboard 目标是让用户快速获得可行动信息，适合使用长度和二维位置等易理解的视觉编码。
2，Fluent UI Sparkline 文档说明 sparkline 适合在空间有限的 dashboard 或 report 中显示随时间变化的趋势。
3，Domo 的 sparkline 指南说明 sparkline 适合表达趋势方向，但不适合承担精确数值读取。
4，收入和支出类流水的精确核对仍应以按时间排序的交易表为准；图表只负责快速展示变化模式。

因此，Codex+++ 首版选择：

1，简洁模式显示剩余 token 精确数字，保证用户第一眼知道还能用多少。
2，简洁模式显示最近 7 日 token 用量迷你趋势图，只用于观察近期消耗趋势，不承载精确核对。
3，高级模式的 token 用量明细保留数字表格，并可用小型横向堆叠条辅助理解 input token、cached input token 和 output token 的组成。
4，高级模式的 token 收入和消耗流水使用最近 30 日正负柱状图快速展示收入和消耗方向，再用流水表提供精确核对。
5，不使用饼图、圆环图、仪表盘或大面积装饰图表达 token；token 可视化必须支持快速、精确地理解数量变化。

号池凭据调研结论
----------------

sub2api 支持多种上游账号类型，包含 OAuth 和 API Key。Codex+++ 首版只围绕 Codex 上游账号建立干净号池，不继承多供应商公开接口形态。

首版号池导入适配以下输入：邮箱密码文本或 JSON、Codex `auth.json`、Token JSON、Sub2API 账号备份和纯 Token 文本。Sub2API 格式只作为账号数据输入，不注册 Sub2API 兼容路由。

首版 Codex 上游账号字段包括：账号名称、账号分组、来源类型、凭据类型、授权状态、`access_token`、`refresh_token`、Codex `auth.json` 快照、临时密码、`token_type`、`expires_at`、可选 `email`、可选 `subscription_tier`、`entitlement_status`、账号状态、余额状态、风控状态、最近授权错误和最近检查时间。

敏感字段必须加密保存，不得在后台列表、审计、日志或普通响应中展示原文。日志最多记录账号 ID、账号名称、分组、状态、脱敏标识或不可逆指纹。

邮箱密码导入后创建待授权候选账号。候选账号不生成内部 API Key，不进入 Gateway 调度。候选密码只保存在服务端加密字段中。授权成功后在同一账号记录上写入授权材料并清除密码密文。

管理员后台使用 Codex app-server 浏览器登录或 device-code 登录完成候选账号授权。浏览器登录的远程回调只接受 loopback 主机、有效端口、`/auth/callback` 路径、匹配的 OAuth state 和授权结果参数。授权 URL、PKCE、令牌交换和刷新由 Codex app-server 管理。

号池导入、候选账号、上游授权、授权刷新和内部 API Key 生成只存在于后端与管理员 Web。普通用户桌面客户端不包含对应接口、页面或本地状态。

号池账号检查首版通过 Codex 官方 app-server 协议实现。后端使用上游账号的 access token 启动 `codex app-server --listen stdio://`，通过 JSON-RPC 调用 `account/read`、`account/rateLimits/read` 和 `account/usage/read`，并把检查结果写回后台可见的账号状态、余额状态、风控状态、计划类型、邮箱和最近检查时间。app-server 传输方式、access token、内部 endpoint 和原始响应不得暴露给普通用户。

号池账号支持批量导入；同一批次可以包含同一种输入格式的多条 Codex 账号。重复邮箱、ChatGPT account id 或凭据指纹更新现有记录。

Windows 配置写入调研结论
------------------------

Codex++ 文档显示 Codex 本机配置位置包括 `~/.codex/config.toml`、`~/.codex/auth.json`，并要求修改 API 模式前确保 `config.toml` 有备份，便于安全回滚。

Codex+++ Windows 与 macOS 客户端允许在启动 Codex 前备份并修改本机 Codex 配置。这里的“回滚”指恢复到 Codex+++ 写入前的本机配置状态，用于启动失败、配置写入失败或用户需要撤销本次自动写入时保护本机环境。

配置回滚不是首屏独立功能，不要求普通用户理解配置文件。首版只在启动失败、配置写入失败或高级模式的诊断区域提供中文动作，例如“恢复启动前配置”。

配置写入必须满足以下约束：

1，写入前创建时间戳备份。
2，使用结构化 TOML 读写，不得用字符串拼接覆盖配置。
3，不得整文件覆盖 `config.toml`。
4，只允许写入或清理 Codex+++ 管理的键和值。
5，必须保留用户已有的插件、MCP、features、desktop、memories、hooks 和 projects 等非 Codex+++ 配置。
6，写入失败或 TOML 解析失败时，不得继续启动，并显示中文错误。
7，必须提供回滚到写入前备份的能力。
8，发现 Codex++、sub2api 或历史实现留下的旧配置项时，应清理为 Codex+++ 唯一干净配置。
9，本机已有认证时，客户端只提交本机识别出的账号标识和认证模式给后端做归属匹配，不提交 token 或 `auth.json`。只有 ChatGPT 账号明确属于本后端号池时才允许继续；账号不属于本后端号池、无法识别，或属于用户自己的 API Key 等其他认证时，不得覆盖登录态或修改配置，只提示用户先退出自己的登录。
10，本机没有 Codex 登录态，或已有登录账号匹配本后端号池时，客户端向后端请求当前普通用户和当前设备专属的 Codex+++ 网关访问 key，保存启动前本机配置快照，并写入 `model_provider = "codexppp"`、`[model_providers.codexppp]`、`wire_api = "responses"`、`base_url = "<backend>/codex/v1"` 和 `requires_openai_auth = true`。该 key 只有在设备启用且短期运行租约有效时可用；停止 Codex 后恢复原快照。
11，允许准备时，客户端写入 Codex `auth.json` 的 `auth_mode = "apikey"` 和 `OPENAI_API_KEY`。客户端写入用户与设备绑定的普通用户网关访问 key，不得写入号池内部路由 API Key；两类 key 必须分表保存，用户 key 不绑定上游账号。上游号池选择、稳定用户绑定、账号活跃用户上限、余额检查、失败切换和扣费仍由后端 gateway 负责。
12，客户端窗口关闭时必须隐藏到系统托盘并继续发送运行心跳；托盘提供重新打开和“停止 Codex 并退出”。明确退出或注销时必须停止本轮受管 Codex、通知后端停止运行租约并恢复启动前快照。异常结束或断网时后端运行租约最多保留两分钟，到期后该设备 key 必须失败关闭；下次启动发现 Codex 已停止时恢复遗留快照。

Windows Store Codex 启动约束
----------------------------

Windows Store 版 Codex 不得按普通命令行程序处理。

已复现失败路径：

1，使用 `cmd /C start` 或等价 shell 二次解析方式启动 `C:\Program Files\WindowsApps\OpenAI.Codex_...\app\Codex.exe`，Windows 弹出“找不到文件 `\codex\`”或等价找不到应用错误。
2，直接 `CreateProcess` 执行 `C:\Program Files\WindowsApps\OpenAI.Codex_...\app\Codex.exe`，客户端显示“启动失败”。

后续实现不得把上述两条路径作为启动方案、重试方案或 fallback。

Windows Store 版 Codex 启动方案发布前必须在部署机完成以下验证：

1，客户端检测到已安装 Codex。
2，客户端点击启动后 Codex 主界面打开。
3，客户端状态刷新为运行中。
4，停止 Codex 后客户端状态刷新为未启动。
5，Codex 读取到 Codex+++ 写入的配置和认证信息。
6，号池账号或等价可用路由在 Codex 中生效。

启动失败诊断必须记录可定位的本机失败阶段。客户端不得只保留 `codex_launch_failed` 这类无法区分失败路径的单一结果。

审计边界
--------

产品审计只记录运行中的管理动作、号池动作、充值确认、token 变动、扣费和异常修正。

编写代码、删除旧代码、清理旧文件、重构实现或迁移仓库内容不是产品运行时动作，不写入产品审计。

Windows 与 macOS 首版范围
-------------------------

Windows 在 Windows 10 / 11 实机验收；macOS 在 macOS 14+ 的 Apple 硬件或运行在 Apple 硬件上的虚拟机验收。

Windows 与 macOS 客户端必须能完成登录、状态展示、发起充值请求、查看 token 余额、本机 Codex 历史配置清理、无本机 Codex 登录态时的 provider 自动配置、一键启动 Codex 和本机 Codex 账号读取。上游 API Key 或等价可用路由切换属于后端 gateway 调度责任，不属于桌面客户端启动前置条件。

Linux、移动端和浏览器插件可以后续扩展，但不得作为首版完成条件。
