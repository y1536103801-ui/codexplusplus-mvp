# Implementation Notes

This implementation follows the current project goals:

- Ordinary users only use the Windows desktop client UI.
- Administrators only use the Web admin console.
- Docker deployment runs backend, administrator console, PostgreSQL, and Redis only; the ordinary-user client remains Windows/Tauri software and is not exposed as a backend-served website.
- Token charging is modeled as a Codex+++ token wallet, RMB-denominated token top-up entries, recharge requests, admin confirmation, token ledger, and usage records.
- Paid token top-up entries are repeatable.
- Free token top-up entries are view-only for ordinary users.
- Codex+++ token shortage never triggers API key switching.
- Upstream Codex account shortage can be handled by selecting another active API key bound to an available upstream account.
- Sensitive upstream credentials are encrypted before persistence and are never returned in list responses.

Reference requirements are explicit in the implementation:

- Codex++ is the reference for the Windows local-companion boundary: the client is a launcher/configuration bridge around local Codex, not a replacement for Codex.
- sub2api is the reference for the operations boundary: upstream account pool, generated platform API keys, gateway forwarding, token-level usage records, and administrator-operated recharge confirmation.
- Codex+++ keeps a unique public API surface. It does not add root-level OpenAI-compatible, Anthropic-compatible, sub2api-compatible, or other old-interface compatibility routes. The authenticated `/api/codex/v1` adapter exists only so local Codex can use the official provider `wire_api = "responses"` path, and it reuses the same gateway billing path.

Component versioning is explicit. Backend and desktop client have separate component ownership. Routine backend commits and routine Windows/macOS-client commits belong to their component Git histories. The integration branch records paired snapshots. `/api/client/...` compatibility is locked by `X-CodexPPP-Interop-Major`; current value is `1`. Both desktop platforms send this header on every client API request. The backend returns the same header and rejects absent or different values with `client_version_incompatible`. A changed `/api/client/...` request or response contract requires a paired backend and desktop-client integration snapshot.

The gateway fails closed on billing. The client request body is sanitized before execution, so client-supplied `usage`, `inputTokens`, `outputTokens`, and related fields are not sent to the upstream and cannot be used for local deduction. The backend deducts only after the upstream response contains usage data. If upstream usage is absent or invalid, no usage record or balance deduction is created.

Gateway turn execution uses the Codex Responses protocol. The backend sends the sanitized request to `https://chatgpt.com/backend-api/codex/responses` with the selected pool account's bearer token and `ChatGPT-Account-ID`, preserves the upstream SSE body and permitted Codex response headers, and extracts usage from the completed response for billing. Local Codex receives tool calls and executes them in the user's local runtime. Backend app-server processes remain limited to account authorization, credential refresh, and account checks.

The Responses ingress owns HTTP request-content decoding. Official Codex can switch larger request bodies to `Content-Encoding: zstd`; Nginx disables request buffering only for the exact Codex Responses route and leaves admission to the authenticated backend. Before consuming the body, the backend applies request-rate and upload-concurrency admission (two uploads globally and one per user by default). It incrementally decodes zstd, validates JSON, removes gateway-only top-level fields, and writes replayable ordinary JSON to a mode-0600 temporary file, so compressed and decoded copies do not accumulate in the Go heap. Encoded and decoded streams each have a 1 GiB product safety ceiling; this ceiling is an engineering guardrail derived from supported multimodal payload sizes, not a published generic OpenAI Responses byte contract. Decoder concurrency and memory are bounded. Body rejection logs contain only path, encoding, content length, status, error code, and decoder cause, never request content or credentials.

Gateway idempotency is part of the billing boundary. Each gateway call uses `Idempotency-Key`, `X-Request-ID`, body `requestId`, or a generated request id. External request ids are normalized before storage, response construction, and audit logging; safe short ids are preserved, and unsafe or overly long ids are replaced with a stable hashed id. Before the upstream call, the backend creates a reserved gateway request against the user's available token balance. Failed calls release the reservation without creating user-visible token ledger rows. Successful calls store the short-lived raw Responses body, content type, permitted response headers, one usage record, and one actual debit ledger row. A later call with the same completed request id returns the stored Responses result and does not call upstream again. PostgreSQL response bodies are never loaded into the resident application snapshot; an exact replay reads only its matching row on demand.

Gateway route switching is also server-owned. The backend builds a list of active API keys whose upstream account is active, balance-available, and risk-available. A normal user remains on the same active upstream across different Codex tasks; each upstream has a configurable simultaneous-user limit (`CODEXPPP_GATEWAY_UPSTREAM_USER_LIMIT`, default 2). The short-lived user assignment is renewed by the managed-client heartbeat and released on stop or lease expiry. On upstream quota exhaustion, authorization failure, rate limiting, invalid route, or service unavailability, the failed upstream account may be replaced by the next candidate route. Upstream responses that reject the request shape are not treated as account-switching events. Codex+++ token shortage never triggers this route-switching path.

Gateway session affinity is durable and shared through Redis when configured. The gateway reads stable session identifiers from Codex session headers or request body fields such as `session_id`, `thread_id`, `conversation_id`, and `codex_session_id`; `previous_response_id` is not treated as a stable task identifier. PostgreSQL stores only the hashed binding with a 30-day expiry, and Redis provides the cross-process fast path. If the bound account is no longer routable or has no available concurrency lease, the normal route candidate order and failure-switching rules apply.

Redis is part of the gateway operations boundary when `CODEXPPP_REDIS_ADDR` is configured. The backend checks Redis during readiness, uses it for per-user request limiting, administrator-login limiting, durable session-affinity lookup, atomic per-upstream concurrency leases with renewal, and a distributed lock for the daily account check. Completed idempotent requests return the stored result without consuming another request-limit slot. If Redis is configured but unavailable during admission or lease acquisition, the backend fails closed and does not call upstream or deduct tokens.

Gateway reservations and settlement use dedicated PostgreSQL transactions. Reservation locks the user wallet and existing reservations; successful settlement atomically changes the idempotency row, user balance, debit ledger, usage record, route timestamps, durable affinity, and audits. A PostgreSQL advisory lock permits only one snapshot-writing backend instance until the remaining low-frequency administrator repositories are fully converted. On startup, reservations left by the previous writer are released with a `backend_restarted` failure and an audit record. While running, an in-flight registry distinguishes a genuine concurrent retry from an orphan record.

Responses SSE is forwarded incrementally and flushed as chunks arrive. Usage is extracted by an incremental SSE parser and settled only after the completion event. At most 2 MiB of a successful upstream body is retained in PostgreSQL for six hours for exact idempotent replay; larger or older successful streams keep the structured completion and usage record without retaining the full raw stream. Completed and failed idempotency metadata is retained for 30 days, audit rows for 180 days, and expired login sessions are removed during startup and hourly maintenance. The low-frequency snapshot writer updates users and idempotency metadata in place, so unrelated administrator mutations do not copy or rewrite stored response bodies.

Admin account ingestion accepts ChatGPT session JSON, email/password text or JSON, Codex `auth.json`, token JSON, Sub2API account-backup objects, and existing plain-token text. ChatGPT session import is a backend-owned normalization path: it extracts the real access token, account id, email, plan, and earliest known expiry; stores only a minimal encrypted auth snapshot; discards unrelated profile fields; marks the credential as short-lived; and updates the existing account and API key when a newer session for the same account id is imported. Email/password input creates a disabled `pending_authorization` upstream candidate. Its password is encrypted in `password_cipher`; no platform API key is created and the gateway does not route the candidate. A successful bound authorization updates the same upstream row, clears `password_cipher`, sets `authorization_status = authorized`, enables the account, and creates its platform API key. Public responses expose only `hasPassword`, source type, authorization status, and a sanitized authorization error code.

Admin account authorization uses Codex app-server managed login. Browser authorization calls `account/login/start` with `type = "chatgpt"`; device-code authorization calls it with `type = "chatgptDeviceCode"`. Both paths run `codex app-server --listen stdio://` in an isolated `CODEX_HOME`, wait for `account/login/completed` plus `account/updated`, read the generated `auth.json`, import refreshable ChatGPT authorization, and remove the temporary Codex home. In remote browser authorization, the admin submits the complete localhost callback URL to `/api/admin/codex-oauth/callback`. The backend accepts only an HTTP loopback host, a valid port, `/auth/callback`, a matching OAuth state, and a callback carrying `code` or `error`, then forwards it to the app-server listener in the backend process namespace. Codex+++ does not store an OpenAI OAuth client id, token endpoint list, or authorization-code exchange implementation.

Upstream auth refresh also uses Codex app-server. The backend writes the stored `auth.json` into an isolated `CODEX_HOME`, starts `codex app-server --listen stdio://`, calls `account/read` with `refreshToken = true`, reads the refreshed `auth.json`, and stores the updated encrypted access token, refresh token, and auth snapshot on the same upstream account. The backend does not call an OpenAI OAuth token endpoint directly for refresh.

The desktop launch path calls `/api/client/launch/prepare` before a managed launch. It sends only the locally recognized account identifier, auth-mode label, and a managed-runtime capability flag, never local tokens or `auth.json`. When a ChatGPT login exists, the backend returns a Codex+++ gateway access key only if the identifier matches a stored pool ChatGPT account ID or email. Any other recognized local authentication, including an unmatched or unrecognized ChatGPT login or an existing personal API key, returns `personal_codex_login_detected`; the client does not call the local preparation command and tells the user to sign out first. A current managed client receives a user-and-device-bound gateway key plus a provisional runtime lease; that key authenticates no upstream account and is accepted only while the same enabled device renews its managed runtime. The first managed launch disables the user's legacy unbound key, while older clients remain usable only until that migration occurs. The response does not return route identifiers, gateway paths, upstream groups, upstream account details, pool credentials, or service-status account display data. The client snapshots the original Codex config and auth files, writes the key into `auth.json` with `auth_mode = "apikey"` and `OPENAI_API_KEY`, configures the Codex+++ provider with `requires_openai_auth = true`, and restores the snapshot when Codex is stopped. Actual route selection, token-balance enforcement, route switching, charging, and idempotency remain inside the backend gateway executor used by `/api/codex/v1/responses`; each successful Codex provider usage record stores the client access-key ID and selected internal route-key ID separately. `/api/gateway/` is not registered as a public route.

Upstream account checks use Codex's app-server protocol rather than a guessed HTTP endpoint. The backend starts `codex app-server --listen stdio://` with `CODEX_ACCESS_TOKEN` set to the encrypted upstream access token, then sends `initialize`, `account/read`, `account/rateLimits/read`, and `account/usage/read` JSON-RPC messages. The result updates internal account-pool state for email, plan type, balance availability, risk availability, entitlement status, and latest check time. While a manual or scheduled check is running, the administrator list returns `checkStatus = checking`; completed failures return `checkStatus = failed` plus a sanitized `checkFailureReason` such as authentication failure, quota exhaustion, rate limiting, timeout, or check-service unavailability. Raw exceptions, paths, tokens, and authorization content are never returned. The protocol transport and access token are never returned to ordinary users.

Current purchasing is a wallet top-up flow, not a subscription-plan system. The fixed entries in the UI are recharge amounts that create pending requests; administrator approval is the only operation that adds tokens to the user balance. This keeps the mechanism compatible with Codex's token-based usage and avoids inventing separate product entitlements.

Administrator user management now separates list operations from user-detail operations. The user list supports search, pagination, status switching, and entry into a selected user's detail view. Manual token adjustment, password reset, that user's token ledger, recharge requests, and usage records live under `/api/admin/users/{id}/...` and are shown in the user detail view. This keeps recharge confirmation on the recharge-request page and keeps manual corrections on the user record.

Account-pool operations are exposed as administrator-only maintenance actions. The upstream account list prioritizes usage context: in-flight Gateway users and running-client leases derived from the user's latest successful Gateway ownership, followed by the latest user after stop or lease timeout, today's routed tokens, upstream remaining percentage or credits, combined availability, and latest check time. Each row shows active-user occupancy against the configured limit. A separate two-minute client-runtime lease is created before managed launch and renewed while the desktop runs; administrators see a pre-task user as active and waiting for first route assignment, without attaching it to a fabricated upstream account. Heartbeats continue while the launcher window is hidden and never accept or return an upstream-account ID, so the client cannot choose or guess pool ownership. Successful usage records retain internal upstream-account and API-key ownership for this aggregation, while ordinary-user usage responses continue to omit those routing fields.

Account-pool export is an administrator-only audited POST action that returns a round-trippable JSON attachment. It intentionally contains decrypted upstream credentials for restoration, but never includes internal platform API keys, key hashes, public prefixes, or credential fingerprints. Browser-login and device-code panels close through a backend cancellation action that terminates the isolated app-server session and restores an unfinished bound candidate to pending authorization. The add-account panel closes locally without submitting its form.

Device management uses the backend pagination contract and keeps the first version limited to viewing devices and enabling or disabling them. It does not expose remote control, diagnostic raw output, scripts, endpoint details, proxy details, or support-workbench behavior.

Ordinary-user visible errors are translated in the desktop client before display. Login failures show only "登录失败"; disabled account and disabled device cases are not distinguished for ordinary users. Launch failures use Chinese messages for local desktop problems such as missing Codex, unreadable configuration, failed configuration write, or failed process launch. During managed preparation, the client adds the official desktop setting `[desktop].localeOverride = "zh-CN"` only when the user has not already selected a language; an explicit locale remains authoritative, and the original configuration is restored on stop. Codex+++ token shortage is not presented as a desktop-launch failure and does not add a recharge prompt or an explicit balance-shortage prompt. The simple-mode service status is returned by `/api/client/me` as only "可用" or "不可用"; it is derived from the user's active state, available Codex+++ token balance, and the existence of at least one routable backend account, but it does not expose route ids, API key ids, upstream ids, or internal reasons.

The ordinary-user advanced panel separates a compact 30-day visualization from record tables. `/api/client/ledger/summary` returns UTC-day buckets with income tokens, expense tokens, and net tokens so the chart shows daily positive and negative token movement instead of the latest raw rows. The ledger table remains a paginated record list, and usage and recharge history use ten rows per page by default. Ordinary-user ledger sources use product-facing labels such as "Token 充值项", "管理员调整", and "Codex token 用量". Safe administrator adjustment remarks are shown as "管理员调整：备注内容"; remarks containing internal routes, local paths, tokens, keys, proxy, endpoint, base URL, upstream details, or other sensitive text are folded back to "管理员调整". Usage detail keeps the required six fields and embeds a small stacked composition bar inside the total-token cell so input, cached input, and output token proportions are visible without adding another product field.

Desktop update checking remains part of the software-client boundary. `/api/client/desktop/update` requires an authenticated ordinary-user session and accepts `platform=windows|macos`; a missing platform keeps the legacy Windows behavior. It returns the current version, latest version, availability, optional release notes, optional SHA-256, and a safe download URL only when deployment metadata is configured. The public website exposes distinct platform downloads through `/api/site/config`, but it does not serve the desktop client UI. After login, the client checks this metadata automatically. When an update is required and Codex is not running, the simple-mode primary action changes from launch to client update. The native Tauri command accepts only an HTTPS `.exe` on Windows or `.dmg` on macOS and a complete SHA-256 digest, enforces a bounded download, and verifies the digest before scheduling the platform installer. Windows runs hidden NSIS `/S`; macOS additionally verifies the app bundle signature and Gatekeeper before replacement. The advanced panel uses the same update action and prevents duplicate triggers while updating; the login page still does not expose update or maintenance controls.

The public website is a purchase-before-login surface, not an ordinary-user web application. It lists only enabled token top-ups with positive price and token amount, and submits manually reviewed account orders to `/api/site/orders`. The buyer contact, optional preferred account, and optional remark are encrypted together at rest; public responses, audit details, and logs do not contain them. The admin console can mark an order contacted or rejected, or fulfill it against an existing active user. Fulfillment credits the order's immutable token snapshot exactly once and writes a token-ledger row. No public registration, automatic payment, payment-success claim, user login, balance lookup, or browser account center is implemented.

The administrator usage and audit page uses the same backend pagination contract as the other list views. It requests ten usage records and ten audit records per page, and the UI exposes separate pagination controls for each list instead of slicing a larger response in the browser. Audit rows include actor role and a resolved actor account when the actor is an administrator or ordinary user, while still excluding Codex credentials, API key material, and recoverable secret fragments.

The administrator usage page also contains account-scoped Token analytics backed by `usage_records`. Its default is the latest-used single upstream account, the latest 30 calendar days in Asia/Shanghai, daily buckets, and average Token per task. Administrators can select up to 20 accounts, choose day/week/month buckets, switch among task count, total Token, and average Token per task, and filter aggregated account-buckets by task-count or total-Token ranges. Within one bucket, a task is the unique combination of upstream account, ordinary user, and persisted Codex session id. A Responses `previous_response_id` is deliberately not treated as a task id because it identifies one preceding response rather than the stable Codex task. Historical records or requests without a stable session, thread, or conversation id fall back to one task per usage record, and the API and UI report the affected record count instead of presenting that estimate as exact session data. The query is limited to 366 calendar days and 5,000 account-bucket points.

The backend supports two persistence modes. With `CODEXPPP_DATABASE_URL`, ordered migrations are recorded in `schema_migrations` with SHA-256 checksums and are never silently rewritten. Gateway billing uses PostgreSQL transactions, while low-frequency administrator mutations remain protected by the single-writer advisory lock. Stored administrator and client session tokens are one-way hashes, not reusable bearer values. `CODEXPPP_SECRET` must be an explicit non-default value and protects encrypted upstream credentials. Without PostgreSQL, local Windows development uses the JSON store and the same application behavior without distributed guarantees.

The Tauri client now owns local Windows/Codex actions:

- Detecting the official ChatGPT desktop app used by Codex on Windows through its `OpenAI.Codex` Store/AppX AppID; the Windows client ignores standalone Codex CLI installations. The Windows app is branded `ChatGPT`, while Codex is its native coding-agent workflow.
- Backing up `~/.codex/config.toml` and `~/.codex/auth.json` before Codex+++ managed writes.
- Removing Codex+++ managed TOML tables and stale Codex+++ provider entries from the user-level `config.toml`.
- Refusing to continue if the existing TOML cannot be parsed.
- Matching an existing local ChatGPT login to this backend's pool before any managed write; unmatched personal logins remain untouched, while matching pool logins are backed up and temporarily switched to the Codex+++ provider.
- Installing the official ChatGPT desktop app for Codex from inside the Codex+++ client, detecting only its `OpenAI.Codex` AppID, and launching it through that AppID. Running-state detection does not require the protected WindowsApps directory to be readable: its primary check uses Windows `tasklist /APPS` and requires the returned package full name to be the official `OpenAI.Codex...__2p2nqsd0c76g0` package. Normal process-path and live package-family checks remain secondary fallbacks. A same-named standalone process is never accepted. Command-line Codex is not an accepted Windows installation or fallback.

Windows Store Codex has two recorded invalid launch paths:

1. `cmd /C start` or an equivalent shell-reparsed launch of `C:\Program Files\WindowsApps\OpenAI.Codex_...\app\Codex.exe`, which has produced a Windows “找不到文件 `\codex\`” or equivalent missing-application dialog.
2. Direct `CreateProcess` execution of `C:\Program Files\WindowsApps\OpenAI.Codex_...\app\Codex.exe`, which has produced a desktop-client “启动失败” state.

Those two paths are not valid startup strategies, retry strategies, or fallbacks for Windows Store Codex. A replacement Windows Store Codex startup path must be validated on the deployment machine for installation detection, Codex window launch, running-state refresh, stop-state refresh, Codex+++ provider token availability, and local Codex account detection before a Windows client installer is treated as usable. Local startup failures must keep a stage-specific diagnostic instead of collapsing every launch failure into `codex_launch_failed`.

The client activates the exact installed `OpenAI.Codex` AppUserModelId through Windows `IApplicationActivationManager` instead of spawning `explorer.exe`. Only a successful activation result with a nonzero, live application process id is an accepted launch; the returned process id is retained for running-state and stop checks. COM initialization, activation-manager creation, activation rejection, a missing process id, and an immediately exited process remain distinct sanitized local failure stages. Once accepted, the client reports `已启动` and sends the ordinary-user launch heartbeat. Other local process inspection continues asynchronously as an enhanced confirmation, but inability to enumerate protected Store processes is not itself a launch failure and must never override a still-live process id returned by Windows activation.

The Windows client has an enabled Tauri bundle configuration targeting NSIS, with `scripts/build-windows-client.ps1` as the repeatable build entry. Without the Tauri CLI, the script builds the Windows executable with `cargo build`; with `-Bundle` and the Tauri CLI installed, it runs the configured Tauri bundle build. This packaging path is separate from Docker and does not create an ordinary-user web service.

The macOS client uses `tauri.macos.conf.json` and `scripts/build-macos-client.sh`. It requires macOS 14+ on Apple hardware and produces app/DMG bundles only after tests pass. Public distribution additionally requires Developer ID signing and Apple notarization. The implementation does not create macOS guests on non-Apple VMware hosts or use hypervisor unlock patches.

Docker product validation covers the server side only. The compose stack starts backend, administrator console, PostgreSQL, and Redis, and the backend image includes the official Codex CLI for account authorization, credential refresh, and account checks. `scripts/verify-codex-app-server.go` verifies the backend container's `codex app-server --listen stdio://` account-maintenance entry over stdio. The ordinary-user client remains the Windows/Tauri executable and is never deployed as a Docker-served website.

The backend defaults to `127.0.0.1:8787` for local Windows development and only accepts loopback traffic unless `CODEXPPP_ADDR` is set explicitly. API CORS does not use a wildcard. It allows same-origin admin requests, the Windows Tauri desktop origins, and local development origins by default; deployment-specific origins must be added through `CODEXPPP_CLIENT_ORIGINS`. The packaged desktop client and its checked-in webview fallback use `https://codex.52cx.top/api`. Release builds keep that address fixed and route client API calls through a restricted native Tauri HTTP command that accepts only `/client/` GET/POST paths. Debug builds may read `CODEXPPP_BACKEND_API_BASE` at build or runtime and normalize it to an `/api` base. This setting is not exposed as an ordinary-user editable endpoint or base URL in the client UI.

The deployment stack keeps `GET /api/health` as process liveness and uses `GET /api/ready` for the container healthcheck. Readiness verifies PostgreSQL and configured Redis connectivity but returns only `{"status":"ready"}`; dependency details, route availability, credentials, and diagnostics are not exposed. Container logs use bounded rotation and all three deployment services use `unless-stopped` restart policies.

Codex login and configuration ownership stays local to the user's machine. During preparation, the Windows client removes stale Codex+++ keys from `~/.codex/config.toml` and stale Codex+++ API-key fields from `~/.codex/auth.json`. A local ChatGPT login must first match this backend's pool by account ID or email. Unmatched or unrecognized personal login state is never modified. With no ChatGPT login, or after a successful pool match, the client receives a user-and-device-scoped gateway key, starts a short runtime lease, snapshots the original files, writes `auth_mode = "apikey"` and `OPENAI_API_KEY` to Codex `auth.json`, and writes `model_provider = "codexppp"`, `[model_providers.codexppp]`, `wire_api = "responses"`, `base_url = "<backend>/codex/v1"`, and `requires_openai_auth = true` to `config.toml`. The key remains independent from the selected upstream account and fails closed when its device lease expires. Stopping Codex restores the snapshot. The client does not use provider `auth.command` as a Codex login mechanism and does not inject `CODEX_HOME` into the Codex process. On Windows, only the installed Store/AppX AppID counts as Codex. Runtime detection accepts any version directory belonging to that official package identity. No Codex package version or release line is hard-coded as “latest” or “compatible”. When WinGet is available, the client first forces a refresh of the official `msstore` source and then queries the full installed-product row for the official Microsoft Store product ID. The query passes only when the returned installed version equals the actual AppX version and no `Available` version is present; an absent or inconsistent product row fails closed. When WinGet is unavailable, the client keeps the version unverified until the user clicks the update action, which runs Microsoft's signed Store Web Installer with `--silent`. Only a successful current-session Store query or Store installer check permits a new managed launch. Capabilities whose official runtime is published only after the first desktop activation are verified after that bootstrap.

Closing the Codex+++ window hides it to the Windows tray instead of ending management. The tray can reopen the launcher or explicitly stop Codex and exit. Explicit client logout and managed application exit stop the managed Codex process and restore the pre-launch configuration snapshot. If the launcher is killed, crashes, or loses power before local cleanup, its device-bound provider key stops being accepted when the two-minute runtime lease expires; on the next launcher start, a stopped Codex session triggers snapshot restoration. Cleanup is gated by the active-session manifest so exiting Codex+++ never terminates an unrelated personal Codex session.

The Windows Store package's protected `app\resources\codex.exe` is not an executable fallback for the launcher: direct execution can fail with access denied, and the official writable runtime under `%LOCALAPPDATA%\OpenAI\Codex\bin\<runtime-id>` may not exist until the desktop app has completed its first activation. The client therefore writes the managed provider configuration first, activates the exact AppID, confirms the returned desktop process, and only then waits briefly for that official runtime. It uses the runtime's official plugin interface to check `browser@openai-bundled`, automatically installs the exact bundled plugin when needed, and verifies it. A separately installed standalone CLI and the protected Store resource are never used as fallbacks. Browser setup failure is shown as a sanitized capability warning without misreporting or rolling back an already running desktop application; the user is directed to the desktop Plugins Directory, never to a manual command or a similarly named third-party plugin.

The desktop client does not start a loopback proxy. The backend exposes only the authenticated Codex-shaped adapter under `/api/codex/v1/models` and `/api/codex/v1/responses`; that adapter delegates to the in-process gateway executor. The Windows desktop launch path only writes Codex configuration to point local Codex at that adapter. The backend still does not expose root-level `/v1/chat/completions`, `/v1/responses`, `/v1/messages`, `/api/gateway/`, Images, Anthropic, sub2api, or historical compatibility routes.
