# Implementation Notes

This implementation follows the current project goals:

- Ordinary users only use the Windows desktop client UI.
- Administrators only use the Web admin console.
- Docker deployment runs backend, administrator console, PostgreSQL, and Redis only; the ordinary-user client remains Windows/Tauri software and is not exposed as a backend-served website.
- Token charging is modeled as a Codex+++ token wallet, RMB-denominated token top-up entries, recharge requests, admin confirmation, token ledger, and usage records.
- Paid token top-up entries are repeatable.
- Free token top-up entries are view-only for ordinary users.
- Codex+++ token shortage never triggers API key switching.
- Upstream Codex account shortage can be handled by selecting another active API key bound to an available upstream account.
- Sensitive upstream credentials are encrypted before persistence and are never returned in list responses.

Reference requirements are explicit in the implementation:

- Codex++ is the reference for the Windows local-companion boundary: the client is a launcher/configuration bridge around local Codex, not a replacement for Codex.
- sub2api is the reference for the operations boundary: upstream account pool, generated platform API keys, gateway forwarding, token-level usage records, and administrator-operated recharge confirmation.
- Codex+++ keeps a unique public API surface. It does not add root-level OpenAI-compatible, Anthropic-compatible, sub2api-compatible, or other old-interface compatibility routes. The authenticated `/api/codex/v1` adapter exists only so local Codex can use the official provider `wire_api = "responses"` path, and it reuses the same gateway billing path.

Component versioning is explicit. Backend and Windows client have separate component ownership. Routine backend commits and routine Windows-client commits belong to their component Git histories. The integration branch records paired snapshots. `/api/client/...` compatibility is locked by `X-CodexPPP-Interop-Major`; current value is `1`. The Windows client sends this header on every client API request. The backend returns the same header and rejects absent or different values with `client_version_incompatible`. A changed `/api/client/...` request or response contract requires a paired backend and Windows-client integration snapshot.

The gateway fails closed on billing. The client request body is sanitized before execution, so client-supplied `usage`, `inputTokens`, `outputTokens`, and related fields are not sent to the upstream and cannot be used for local deduction. The backend deducts only after the upstream response contains usage data. If upstream usage is absent or invalid, no usage record or balance deduction is created.

Gateway execution uses Codex app-server as the only runtime path. The backend starts `codex app-server --listen stdio://` with `CODEX_ACCESS_TOKEN`, initializes the JSON-RPC connection, creates a thread with `thread/start`, starts a turn with `turn/start`, and waits for completion notifications. No bridge run URL, root-level OpenAI-compatible route, sub2api route, or historical fallback is part of the runtime contract.

Gateway idempotency is part of the billing boundary. Each gateway call uses `Idempotency-Key`, `X-Request-ID`, body `requestId`, or a generated request id. External request ids are normalized before storage, response construction, and audit logging; safe short ids are preserved, and unsafe or overly long ids are replaced with a stable hashed id. Before the upstream call, the backend creates a reserved gateway request against the user's available token balance. Failed calls release the reservation without creating user-visible token ledger rows. Successful calls convert the request to completed, create one usage record, and create one actual debit ledger row. A later call with the same completed request id returns the stored usage record and does not call upstream again.

Gateway route switching is also server-owned. The backend builds a list of active API keys whose upstream account is active, balance-available, and risk-available. On upstream quota exhaustion, authorization failure, rate limiting, invalid route, or service unavailability, the failed upstream account is marked unavailable and the gateway attempts the next candidate route. Upstream responses that reject the request shape are not treated as account-switching events. Codex+++ token shortage never triggers this route-switching path.

Redis is now part of the gateway operations boundary when `CODEXPPP_REDIS_ADDR` is configured. The backend checks Redis during startup and uses it for per-user gateway request limiting before a new upstream call is reserved or executed. Completed idempotent gateway requests return the stored result without consuming another Redis rate-limit slot. If Redis is configured but unavailable during a new gateway call, the backend fails closed with `rate_limiter_unavailable` and does not reserve, route, call upstream, or deduct tokens.

The desktop launch path calls `/api/client/launch/prepare` only when local Codex has no ChatGPT login. That response returns a backend generated Codex API key for the current user device. It does not return route identifiers, gateway paths, upstream groups, upstream account details, or pool credentials. The client writes the key into Codex `auth.json` with `auth_mode = "apikey"` and `OPENAI_API_KEY`, then configures the Codex+++ provider with `requires_openai_auth = true`. Actual route selection, token-balance enforcement, route switching, charging, and idempotency remain inside the backend gateway executor used by `/api/codex/v1/responses`; `/api/gateway/` is not registered as a public route. When Codex+++ token is unavailable, the gateway stops route supply for the Codex request instead of making the desktop launcher show a balance-shortage prompt.

Upstream account checks use Codex's app-server protocol rather than a guessed HTTP endpoint. The backend starts `codex app-server --listen stdio://` with `CODEX_ACCESS_TOKEN` set to the encrypted upstream access token, then sends `initialize`, `account/read`, `account/rateLimits/read`, and `account/usage/read` JSON-RPC messages. The result updates internal account-pool state for email, plan type, balance availability, risk availability, entitlement status, and latest check time, while the administrator-facing response exposes only the account's combined availability plus the one-time check summary. The protocol transport and access token are never returned to ordinary users.

Current purchasing is a wallet top-up flow, not a subscription-plan system. The fixed entries in the UI are recharge amounts that create pending requests; administrator approval is the only operation that adds tokens to the user balance. This keeps the mechanism compatible with Codex's token-based usage and avoids inventing separate product entitlements.

Administrator user management now separates list operations from user-detail operations. The user list supports search, pagination, status switching, and entry into a selected user's detail view. Manual token adjustment, password reset, that user's token ledger, recharge requests, and usage records live under `/api/admin/users/{id}/...` and are shown in the user detail view. This keeps recharge confirmation on the recharge-request page and keeps manual corrections on the user record.

Account-pool operations are exposed as administrator-only maintenance actions. The upstream account list shows group, combined availability, and latest check time; administrators can check an account and enable or disable it without exposing credentials. The account availability action accepts only the combined `availabilityStatus` value. Balance and risk availability remain internal routing inputs and are folded into the combined availability field, not exposed as administrator API request or response fields. API keys generated from upstream accounts can be enabled or disabled from the same account-pool page. API key raw values, stable key prefixes, key hashes, upstream split status fields, and upstream credential fingerprints remain internal state and are not returned in admin list, creation, check, or audit responses.

Device management uses the backend pagination contract and keeps the first version limited to viewing devices and enabling or disabling them. It does not expose remote control, diagnostic raw output, scripts, endpoint details, proxy details, or support-workbench behavior.

Ordinary-user visible errors are translated in the desktop client before display. Login failures show only "登录失败"; disabled account and disabled device cases are not distinguished for ordinary users. Launch failures use Chinese messages for local desktop problems such as missing Codex, unreadable configuration, failed configuration write, or failed process launch. Codex+++ token shortage is not presented as a desktop-launch failure and does not add a recharge prompt or an explicit balance-shortage prompt. The simple-mode service status is returned by `/api/client/me` as only "可用" or "不可用"; it is derived from the user's active state, available Codex+++ token balance, and the existence of at least one routable backend account, but it does not expose route ids, API key ids, upstream ids, or internal reasons.

The ordinary-user advanced panel separates a compact 30-day visualization from record tables. `/api/client/ledger/summary` returns UTC-day buckets with income tokens, expense tokens, and net tokens so the chart shows daily positive and negative token movement instead of the latest raw rows. The ledger table remains a paginated record list, and usage and recharge history use ten rows per page by default. Ordinary-user ledger sources use product-facing labels such as "Token 充值项", "管理员调整", and "Codex token 用量". Safe administrator adjustment remarks are shown as "管理员调整：备注内容"; remarks containing internal routes, local paths, tokens, keys, proxy, endpoint, base URL, upstream details, or other sensitive text are folded back to "管理员调整". Usage detail keeps the required six fields and embeds a small stacked composition bar inside the total-token cell so input, cached input, and output token proportions are visible without adding another product field.

Desktop update checking is part of the Windows software client boundary, not a public web surface. `/api/client/desktop/update` requires an authenticated ordinary-user session and returns the current version, latest version, availability, optional release notes, optional SHA-256, and an `http` or `https` download URL only when deployment metadata is configured. Docker passes these metadata variables only to the backend API. The advanced-mode client panel displays the update state and uses a Tauri command to open the update download from the desktop shell; the login page does not expose update or maintenance controls, and the backend does not serve the ordinary-user client UI as a website.

The administrator usage and audit page uses the same backend pagination contract as the other list views. It requests ten usage records and ten audit records per page, and the UI exposes separate pagination controls for each list instead of slicing a larger response in the browser. Audit rows include actor role and a resolved actor account when the actor is an administrator or ordinary user, while still excluding Codex credentials, API key material, and recoverable secret fragments.

The backend now supports two persistence modes. When `CODEXPPP_DATABASE_URL` is set, it applies the PostgreSQL migrations and persists runtime state to PostgreSQL. In that mode, `CODEXPPP_SECRET` must be explicitly set to a non-default value before startup because it protects encrypted upstream credentials and session signing. When `CODEXPPP_DATABASE_URL` is not set, the backend uses the local JSON store and may fall back to the development secret so the project can still run immediately on Windows without requiring a database service.

The Tauri client now owns local Windows/Codex actions:

- Detecting local Codex installation and distinguishing command-line Codex from Windows Store Codex.
- Backing up `~/.codex/config.toml` and `~/.codex/auth.json` before Codex+++ managed writes.
- Removing Codex+++ managed TOML tables and stale Codex+++ provider entries from the user-level `config.toml`.
- Refusing to continue if the existing TOML cannot be parsed.
- Preserving existing local ChatGPT login, or configuring official Codex API-key auth and the Codex+++ provider when local Codex has no ChatGPT login.
- Launching local Codex through a verified Windows launch path.

Windows Store Codex has two recorded invalid launch paths:

1. `cmd /C start` or an equivalent shell-reparsed launch of `C:\Program Files\WindowsApps\OpenAI.Codex_...\app\Codex.exe`, which has produced a Windows “找不到文件 `\codex\`” or equivalent missing-application dialog.
2. Direct `CreateProcess` execution of `C:\Program Files\WindowsApps\OpenAI.Codex_...\app\Codex.exe`, which has produced a desktop-client “启动失败” state.

Those two paths are not valid startup strategies, retry strategies, or fallbacks for Windows Store Codex. A replacement Windows Store Codex startup path must be validated on the deployment machine for installation detection, Codex window launch, running-state refresh, stop-state refresh, Codex+++ provider token availability, and local Codex account detection before a Windows client installer is treated as usable. Local startup failures must keep a stage-specific diagnostic instead of collapsing every launch failure into `codex_launch_failed`.

The Windows client has an enabled Tauri bundle configuration targeting NSIS, with `scripts/build-windows-client.ps1` as the repeatable build entry. Without the Tauri CLI, the script builds the Windows executable with `cargo build`; with `-Bundle` and the Tauri CLI installed, it runs the configured Tauri bundle build. This packaging path is separate from Docker and does not create an ordinary-user web service.

Docker product validation covers the server side only. The compose stack starts backend, administrator console, PostgreSQL, and Redis, and the backend image includes the official Codex CLI for gateway execution. `scripts/verify-codex-app-server.go` verifies the backend container's `codex app-server --listen stdio://` entry over stdio. When no real Codex access token is provided, the script stops after a successful `initialize` check. When `CODEXPPP_VERIFY_CODEX_ACCESS_TOKEN` is set, it can additionally verify `account/read`, `account/rateLimits/read`, `account/usage/read`, and an optional real turn through `-turn-prompt`. The ordinary-user client remains the Windows/Tauri executable and is never deployed as a Docker-served website.

The backend defaults to `127.0.0.1:8787` for local Windows development and only accepts loopback traffic unless `CODEXPPP_ADDR` is set explicitly. API CORS does not use a wildcard. It allows same-origin admin requests, the Windows Tauri desktop origins, and local development origins by default; deployment-specific origins must be added through `CODEXPPP_CLIENT_ORIGINS`. The desktop client reads the backend API base from `CODEXPPP_BACKEND_API_BASE` at build or runtime and normalizes it to an `/api` base. If the variable is absent, it falls back to `http://localhost:8787/api` for local Windows operation. This deployment setting is not exposed as an ordinary-user editable endpoint or base URL in the client UI.

The deployment stack includes a backend liveness check at `GET /api/health`. It is intentionally limited to `{"status":"ok"}` and does not expose database state, Redis state, route availability, upstream account status, API key material, credential fingerprints, or diagnostic internals.

Codex login and configuration ownership stays local to the user's machine. During preparation, the Windows client removes stale Codex+++ keys from `~/.codex/config.toml` and stale Codex+++ API-key fields from `~/.codex/auth.json`. If local Codex already has ChatGPT tokens, preparation keeps that login and removes stale Codex+++ API-key state. If local Codex has no ChatGPT login, the client writes `auth_mode = "apikey"` and `OPENAI_API_KEY` to Codex `auth.json`, and writes `model_provider = "codexppp"`, `[model_providers.codexppp]`, `wire_api = "responses"`, `base_url = "<backend>/codex/v1"`, and `requires_openai_auth = true` to `config.toml`. The client does not use provider `auth.command` as a Codex login mechanism. The client does not inject `CODEX_HOME` into the Codex process, and does not display backend pool-account data as the local Codex account.

The desktop client does not start a loopback proxy. The backend exposes only the authenticated Codex-shaped adapter under `/api/codex/v1/models` and `/api/codex/v1/responses`; that adapter delegates to the in-process gateway executor. The Windows desktop launch path only writes Codex configuration to point local Codex at that adapter. The backend still does not expose root-level `/v1/chat/completions`, `/v1/responses`, `/v1/messages`, `/api/gateway/`, Images, Anthropic, sub2api, or historical compatibility routes.
